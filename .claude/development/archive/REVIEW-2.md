## 1. Позиционирование и актуальность — 7.5/10

> **Ревизия 2026-06-04 (после возражений владельца + чтения STRATEGY-3.md).** Первая редакция
> ставила 6.5 и сравнивала Beadloom с *коммерческими* point-инструментами как с заменами. Это
> была ошибка рамки: (а) пропущена ось **MIT/self-hosted/zero-cost**, (б) недооценён **агентный
> loop сопровождения** (AI-техпис в CI = роадмап F4.1, не «идея ревьюера»), (в) портал задешевлён
> как «ответ Backstage». Ниже — исправленная оценка. Критичные оговорки сохранены.

Тайминг отличный: проект стоит ровно на горячих трендах 2026 (context engineering вытеснил
prompt engineering; MCP в декабре 2025 ушёл под Linux Foundation, 97M загрузок/мес — ставка на
MCP безопасна; «архитектурный дрейф» от агентного кода — острая боль).

**Ключевая поправка — ось лицензии.** Если разложить пересекающиеся инструменты по
лицензии/охвату, открывается незанятая позиция:

| Дают часть ценности Beadloom | Лицензия | Охват / ограничение |
|---|---|---|
| Swimm, Sourcegraph Amp, CodeScene, Port, Apollo GraphOS (платформа) | **Commercial / per-seat / SaaS** | кросс-срезы, но платно + lock-in |
| Backstage | OSS (Apache) | catalog, ~$150K TCO, <10% adoption, не enforcement |
| Codemap (17 MCP-tools), CodeGraph, Understand-Anything (15k★) | **MIT/OSS** | **single-repo** (CodeGraph явно «no federation») |
| ArchUnit | OSS | **только JVM** (Beadloom — полиглот, 11 языков) |
| Pact, Apollo GraphOS, Buf | OSS-CLI / commercial | **один протокол**, без единого графа ландшафта |
| repomix / gitingest / Context7 | OSS | плоский дамп, без графа/правил |

**Вывод:** свободного MIT-инструмента, дающего связку **«federated contracts + полиглот-enforcement
+ doc-sync + авто-KB-портал + агентное сопровождение»**, не существует. OSS-аналоги либо single-repo,
либо один протокол, либо JVM-only, либо плоский дамп; всё кросс-срезовое — коммерческое.

**Три защищаемых актива (не «фичи», а связка):**
1. **Кросс-репо federation граф контрактов микросервисов** (AMQP/GraphQL) — все ближайшие
   graph-for-agents single-repo by design; Pact/Buf/Apollo точечны на один протокол.
2. **Замкнутый агентный loop сопровождения** (STRATEGY-3 F4.1) — `sync-check` ловит дрейф →
   AI-техпис чинит ТОЛЬКО дрейфнувшие узлы (scoped, дёшево) → re-check до фикспоинта → PR на
   ревью; tool-agnostic, external-model, OSS-совместимо. Это «агенты поддерживают базу знаний
   сами», а не «context для агента» (последнее коммодитизируется). Loop detection→remediation→
   verification у point-инструментов отсутствует.
3. **Авто-портал honest-by-construction** — каждая цифра дашборда из того же code path, что и
   гейт (дашборд физически не может разойтись с `beadloom ci`); badge свежести из `doc_sync`;
   регенерация из графа на каждый push (zero-maintenance). Дифференциатор не «как Backstage», а
   «не дрейфует и бесплатно».

**Интеграция и есть ценность.** Это не 4 разрозненных бета, а один loop (граф → enforcement →
doc-sync → агентная починка → портал). Ни один point-tool не даёт весь loop бесплатно и
self-hosted — это и есть позиция.

**Где НЕ капитулирую (остаюсь критичным):**
- «Бесплатно» — ось *дистрибуции/adoption*, не защита от *следующего* OSS-входящего. Защищает
  интеграция loop, а не цена сама по себе.
- **Capacity-риск одного мейнтейнера** на 5 фронтов OSS *усиливается* (нет выручки нанять людей) —
  это собственный §6 STRATEGY-3, «#1 risk».
- Портал ≠ Backstage-killer: статический KB без live-каталога/ownership/scorecards/плагинов.
  Другая ценность, не замена.
- **presence-check контрактов** остаётся методологически слабее Pact/Buf (наличие узла ≠
  семантическая совместимость) — но как OSS landscape-level early-warning это честно и уникально.

**Риски на 12-18 мес (пересмотрено):**
1. IDE/агенты съедают *single-repo* контекст-слой — высокий (но не federation/loop).
2. OSS graph-for-agents добавит federation и сотрёт преимущество #1 — средне.
3. presence-check vs Pact/Buf — методологически слабее (митигируется честным позиционированием).
4. Один мейнтейнер на 5 фронтов — **главный риск** (распыление + нет выручки).

**Рекомендации по заострению позиционирования:**
- Вести нарратив как **«free, self-hosted, MIT federation+governance loop для микросервисов в
  эпоху агентов»** — явно противопоставляя per-seat/SaaS-конкурентам и тяжёлому Backstage.
- Продавать **связку loop**, а не отдельные фичи: «один MIT-инструмент вместо Swimm + Amp +
  GraphOS + Backstage».
- Позиционировать как слой НАД IDE (соседний сервис + контракт), не конкурент single-repo контексту.
- Контрактную часть — честно как «landscape-level early warning», план на семантику (Buf/Apollo как источник).
- Дистрибуция через MCP-реестр + интеграции с Claude Code/Cursor (federation-MCP-сервер).

## 2. Качество кода и архитектуры — 7.5/10

**Сильное:** статанализ чист, безопасность продумана (subprocess только списком,
`yaml.safe_load`, git-ref валидируется, 0 eval/shell=True/мутабельных дефолтов/bare-except),

Context Oracle реально держит «<20ms» (DB-индекс + L1-кэш с etag).

**Проблемы (реальные, файл:строка):**

1. **[HIGH] Нет слоя доступа к данным.** Сырой `conn.execute("SELECT…")` размазан по
   **35 файлам** во всех 7 доменах, включая `tui/data_providers.py:36-42` — презентация
   напрямую читает SQLite (протечка слоя). Один запрос `SELECT ref_id, kind, summary FROM nodes`

   → ввести `infrastructure/repository.py`; `tui` ходит только через application-слой.
2. **[HIGH] 53 вызова `open_db` и ни одного `with`/`contextlib.closing`.** Утечка соединений

   багов. Подтверждено тестами: 589 ResourceWarning «unclosed database».
   Примеры: `application/reindex.py`, `services/cli.py` (~6 ручных close в `status`).
3. **[HIGH] N+1 в `doc_sync/engine.py:451-525`** (`check_source_coverage`): вложенные запросы

   большом графе. → префетч одним JOIN/`IN (...)`; нормализованная таблица аннотаций.
4. **[HIGH] Детекция циклов `rule_engine.py:1170-1209`** (`evaluate_cycle_rules`) — DFS из

5. **[MEDIUM] God-домены реальны, а не ложный линт.** `graph/` (6439 строк, rule_engine 1743 +

6. **[MEDIUM] God-функции:** `cli.py:status` ~283 строки L800-1072 (бизнес-логика в CLI-слое →

## 5. Killer-features — рекомендации

Отсортировано по стратегической ценности (заострение защищаемой ниши, не распыление).

**Tier 1 — углубляют federation-ров (то, чего нет ни у кого):**
1. **Семантические контракты вместо presence-check.** Главное слабое звено. Парсить реальную
   AMQP-схему / GraphQL SDL / AsyncAPI/OpenAPI и сравнивать *поля и типы*, а не наличие узла.
   Интегрировать Buf/Apollo как источник истины. Превращает «early-warning» в настоящий
   contract-testing на уровне ландшафта — единственный кросс-протокольный и кросс-репо.
2. **Federation-MCP-сервер.** Подавать агенту контекст *соседнего* сервиса и контракта между
   репозиториями через MCP. Ровно то, что single-repo индекс Cursor/Amp не видит.
   Заход через MCP-реестр как «cross-repo context server».
3. **«Blast radius» по ландшафту.** `beadloom why <contract> --landscape` — кто сломается во
   всех репозиториях при изменении контракта/узла. Impact через границы репо — киллер для микросервисов.

**Уже на роадмапе (STRATEGY-3, не дублировать — приоритизировать):**
- **AI-техпис в CI (F4.1)** — замкнутый loop scoped-починки дрейфнувших доков, re-check до
  фикспоинта, PR на ревью. Это actually killer-feature #2 (см. §1); спроектирован, отложен. Не
  «рекомендация ревьюера», а следующий эпик. → довести до релиза первым из агентных фич.

**Tier 2 — усиливают intent-vs-reality:**
4. **Architecture drift over time.** Уже есть `snapshot` + `metrics_history`. Добавить «decay
   report»: появившиеся вопреки правилам зависимости, рост cycles, домены пробивающие лимит — тренд.
5. **PR-bot / GitHub App.** Инлайн-комментарий в PR: «добавлен edge X→Y, нарушающий layer-rule» /
   «контракт Z стал BREAKING для @backend». Сейчас enforcement только в CI exit-code; инлайн
   кратно поднимает adoption. Естественно сочетается с агентным loop F4.1 (агент чинит → PR-bot
   объясняет).
6. **Auto-bootstrap графа из кода (довести до ума).** Сейчас помечен как work-in-progress. Это и
   есть ответ Codemap/CodeGraph: гибрид «выведено из кода + намеренческий слой поверх».
   Без этого ручной YAML — барьер входа.

**Tier 3 — гигиена/долг (разблокируют рост):**
7. Repository-слой + контекст-менеджеры соединений (чинит HIGH #1 и #2 разом).
8. Реальный движок диаграмм ландшафта (Cytoscape/D3 вместо Mermaid) для крупных ландшафтов.

## 6. Глубокий разбор двух слабых мест (с оценкой реализации)

> Добавлено 2026-06-04 по запросу владельца. Grounded в реальном коде: `graph/contracts.py`,
> `graph/sdl.py`, поля узла в `graph/loader.py:42` (`_NODE_DIRECT_FIELDS = {ref_id, kind,
> summary, source, lifecycle}` + произвольный `extra` JSON).

### 6.1 Presence-check контрактов — методологически слабее Pact/Buf

**Что сейчас в коде.** `ContractVerdict.BREAKING` живёт в одном месте — `Contract.missing_references`
(`contracts.py:124`): `sorted(set(self.references) - set(self.exposed))`.
- `exposed` — множество **имён**, которое `sdl.py` извлекает регуляркой: имена полей
  `Query/Mutation/Subscription` + имена top-level `type/input/enum/interface`. Это
  **не парсер**, а строчно-скобочный сканер (сам docstring `sdl.py` помечает `graphql-core`
  как upgrade path).
- `references` — **рукописный список** имён из payload ребра (`_accumulate_surface`), НЕ
  извлечённый из реальных запросов consumer'а.
- Для **AMQP `missing_references` всегда `[]`** (`if self.protocol != _GRAPHQL`) → для очередей
  BREAKING физически невозможен, только presence продюсера+консьюмера = CONFIRMED.

Итог: отвечаешь на **«осталось ли ИМЯ?»**. Pact/Buf/GraphOS отвечают на **«совместима ли
ФОРМА/ТИП/wire-формат с реально используемым?»**. Presence ⊂ semantic.

**Ловит честно:** удаление поля/типа целиком, переименование (имя пропадает → BREAKING). ✅

**Пропускает (ложный CONFIRMED):**

| Изменение продюсера | Рантайм consumer'а | Beadloom сейчас |
|---|---|---|
| `user: User` → `user: ID` (смена типа) | `user { name }` падает | **CONFIRMED** ❌ |
| `email: String!` → `email: String` (nullability) | NPE на non-null | не видит ❌ |
| добавлен **обязательный** аргумент `users(active: Boolean!)` | старые запросы отклоняются | не видит (regex берёт идентификатор до `(`) ❌ |
| убрано значение enum `Role.USER` | запрос с `USER` падает | не видит (берутся имена *типов*, не значения) ❌ |
| удалено `User.email` (поле НЕ top-level операции) | `me { email }` падает | **не может проверить** (`sdl.py` не извлекает поля обычных типов) ❌ |
| AMQP: из тела сообщения убрано поле / сменён тип | десериализация падает | **CONFIRMED** (для AMQP surface нет) ❌ |

Доп. зазор: `references` рукописный → если разойдётся с фактическим кодом consumer'а, ложная
уверенность с обеих сторон.

**Как делают специалисты:** Pact — consumer-driven, проигрывает *реальные* интеракции против
provider'а в CI (+ `can-i-deploy`); Buf — парсит полный protobuf-дескриптор, классифицирует
breaking по категориям (WIRE/WIRE_JSON/PACKAGE/FILE); Apollo GraphOS — валидирует схему против
**исторических операций реальных клиентов** на уровне полей/аргументов.

**Стоит ли строить — три уровня:**
- **Tier A — настоящий GraphQL-парсинг** (`graphql-core` за optional-extra `[graphql]`,
  ядро держать dep-free). Извлекать field-level surface (`Type.field: ReturnType!` + args),
  `classify` делает реальный schema-diff (удалённое поле, смена/сужение типа, новый
  required-аргумент, non-null на входе, удалённое значение enum). **Самый высокий рычаг за
  наименьшую цену; убирает худшие ложные CONFIRMED. Стоимость: средняя.**
- **Tier B — извлекать `references` из реальных запросов** (`.graphql`, `gql\`\`` в TS, `gql`
  в Python) → сравнение фактического использования против реальной схемы ≈ GraphOS
  operation-checks. **Стоимость высокая (по-языковая экстракция); вероятно не строить до Tier C.**
- **Tier C — НЕ переизобретать, а федерировать вердикты специалистов (это и есть ров).**
  `buf breaking` (gRPC) / `graphql-inspector`/Apollo (GraphQL) / Pact-broker `can-i-deploy`
  (HTTP) отдают вердикт → Beadloom *проглатывает* его как `ContractVerdict` ребра.
  Presence-check остаётся честным zero-config fallback'ом. Beadloom — **единственный, кто сводит
  Pact + Buf + GraphOS + AMQP в ОДИН ландшафт с ОДНИМ гейтом**, ни один из них не видит соседние репо.

**Рекомендация:** Tier A (дешёвая победа) + для AMQP — diff схемы тела сообщения, если есть
задекларированный источник (AsyncAPI/JSON-Schema в `openspec/specs`) + Tier C как настоящий ров
(«принеси свой Pact/Buf — мы федерируем вердикт»). Tier B пометить дорогим и отложить.

### 6.2 Портал ≠ Backstage — чего нет и что строить

**Чего нет (каждая строка — настоящий gap):**

| # | Нет в Beadloom | Почему важно |
|---|---|---|
| 1 | **Каталог с ownership** (owner/team/system) | Federation поднимает вопрос №1: «**кого звать**, когда контракт `@backend:X` сломался?» Сейчас ответа нет — узел несёт только `ref_id/kind/summary/source/lifecycle` + `extra` |
| 2 | **Scorecards / tech-health** (on-call? runbook? coverage? SLO? vulns?) | Backstage Soundcheck / Port — зрелость во времени, gate «production readiness». У Beadloom есть debt+lint, но это арх-правила, не широкий операционный скоринг |
| 3 | **Live-данные / плагины** (CI, PagerDuty, Sonar, k8s, cloud) | Портал — статический снапшот, между билдами устаревает, плагин-модели нет |
| 4 | **Scaffolder / golden paths** (новый сервис из шаблона) | killer-фича Backstage; у Beadloom только `init --bootstrap` для существующего |
| 5 | **Self-service + RBAC** (триггерить деплой, права) | Beadloom read-only + CI-гейт |

**Где портал ОБЪЕКТИВНО лучше (не потерять):** honest-by-construction (цифры из того же code path,
что и гейт — не дрейфуют, в отличие от каталога/TechDocs Backstage); zero-maintenance (регенерация
из графа); free / no infra (vs $150K TCO + живой Node-app); арх-правда (контрактный landscape +
enforcement Backstage не делает вообще).

**Матрица решения (честно):**
- **Ownership — ДА, дёшево, по стратегии.** `owner`/`team` (+ опц. `system`) как first-class поле
  узла. Питает уведомление владельцу при BREAKING, PR-bot, дашборд. Стоимость: низкая.
  **Оговорка:** ownership дрейфует (то, за что критикуем Backstage) → выводить из **CODEOWNERS /
  git blame**, а `sync-check`-подобный детект флажит owner-vs-реальность. Тогда остаётся
  honest-by-construction, а не гниющий `catalog-info.yaml`.
- **Scorecards — ЧАСТИЧНО, как «арх/governance-scorecard».** Входы уже есть (lint, debt,
  doc-freshness %, контрактные вердикты, циклы, coverage) → per-service readiness-rollup
  («граф чист? доки свежи? нет breaking? границы соблюдены?»), honest-by-construction.
  **НЕ тянуть** Sonar/PagerDuty/SLO/vuln — это live-plugin-нора и убийца capacity.
- **Live / плагины / scaffolder / self-service / RBAC — НЕТ.** Другой продукт, фора Backstage/Port
  в 6 лет, гарантированная смерть от scope для одного мейнтейнера (§6 STRATEGY-3). Статический
  честный снапшот — фича, не дефект. Если live-интеграция понадобится — **обратный ход**: Beadloom
  эмитит `catalog-info.yaml` / Backstage-entity export, и оргструктура, которая *уже* крутит
  Backstage, поглощает граф Beadloom как источник арх-правды. **«Beadloom кормит Backstage», не «заменяет».**

### 6.3 Общий принцип

В обоих случаях выигрышный ход один и совпадает с рвом: **Beadloom — честный
агрегатор/федератор, а не переизобретатель специалиста.** Контракты — федерируй Pact/Buf/GraphOS
(Tier C), presence оставь fallback'ом, для GraphQL сделай дешёвый Tier A. Портал — добавь
ownership (из CODEOWNERS) + арх-scorecard и **корми** Backstage, не конкурируй за live-портал.
Никто не владеет **кросс-репо ландшафтом + единым гейтом + honesty** — туда и углублять.

**Самые дешёвые высокоэффективные первые шаги:** (а) owner-поле из CODEOWNERS + drift-check;
(б) GraphQL Tier A через `graphql-core` за extra; (в) verdict-ingestion адаптер (Tier C) для
одного протокола как PoC.

## 7. «Beadloom AI-multi-agentics flow» — оценка мультиагентного процесса

> Добавлено 2026-06-04 по запросу владельца. Прочитаны: `.claude/CLAUDE.md`, все 4 `.claude/agents/*.md`
> (dev/test/review/tech-writer), `.claude/commands/*` (coordinator 236 строк, task-init, checkpoint,
> templates), и реальный эпик BDL-040 (PLAN+ACTIVE). Git-история подтверждает исполнение флоу
> (коммиты помечены `W1/W2/W3`, `test wave`, `review wave`, по бедам) — это работающий процесс,
> а не аспирационная документация. Он произвёл кодовую базу с 91.6% покрытия, mypy-strict, 0 TODO.

**Оценка как инженерии процесса: 8/10.** Лучше подавляющего большинства agentic-сетапов.

### Что реально сильно
1. **Context engineering — на голову выше типичного.** Бьёт в главный failure mode мультиагентной
   разработки (переполнение контекста + потеря состояния при компакции) системно:
   - **Coordinator Context Boundary** (`NEVER reads src/**, tests/**`) — ключевой инсайт: координатор
     остаётся маленьким, делегируя всё чтение кода суб-агентам. Большинство этого не делают → их
     оркестратор раздувается на длинном прогоне.
   - **Return contracts** (2-3 строки, детали в bead-comments) — вывод суб-агентов не загрязняет родителя.
   - **Файловая память** (CONTEXT/ACTIVE/PLAN + bead-comments) против авто-компакции.
2. **DAG-как-гейты** — downstream-бед не появляется в `bd ready` пока не закрыты блокеры; волны
   вытекают из DAG; `review→ISSUES→fix-cycle` петля; `merge-slot` для сериализации landings.
3. **Зашитая честность, выстраданная** — в `review.md`/`tech-writer.md` закодирован тонкий гочча
   («sync-check показывает `[ok]` даже при устаревших доках, т.к. reindex сбрасывает baseline →
   кросс-чекай API CHANGE + grep docs/»), re-check-to-fixpoint, «accuracy over volume, mark unknowns».
4. **Продукт И ЕСТЬ субстрат флоу — замкнутый контур.** Агенты берут контекст через `beadloom
   ctx/why/prime`, верифицируются через `lint/sync-check`, tech-writer поддерживает тот самый
   граф/доки, что кормят следующих агентов. Это сочетание действительно редкое.

### Где честно критично
1. **Это процесс, а не ров — и он копируемый.** ~1100 строк Markdown (commands) + 136 (agents) +
   CLAUDE.md, поверх Claude Code + `bd` (steveyegge/beads, внешний). Почти всё —
   prompt-engineering + конвенция, не код Beadloom. Любой может скопировать `.md`. **Защищает не
   флоу, а его интеграция с графом Beadloom** (ctx/why/lint/sync-check как субстрат агента).
2. **Зависимость от двух чужих движущихся субстратов** — `bd` (swarm/gate/merge-slot) и Agent-tool
   Claude Code. Ни тем, ни другим владелец не управляет; уже логируются `bd`-квирки (UX #97).
   Поменяется их API/семантика — флоу хрустнет.
3. **Откалибровано под n=1 (один человек + один проект).** Approval-гейты на каждом из
   PRD/RFC/CONTEXT/PLAN предполагают одного human-in-the-loop владельца. Это
   **solo-maintainer-with-AI-fleet паттерн**, НЕ «team multi-agent development». Прямое напряжение с
   заявленной целью «внутренний командный инструмент»: на команду из N параллельных владельцев
   последовательный координатор-на-одобрениях не масштабируется.
4. **Много церемонии — честный вопрос об источнике качества.** Сколько даёт *церемония*, а сколько
   *Opus + TDD + строгие гейты*? Скорее второе. Ценность церемонии — выживание контекста на
   ДЛИННЫХ прогонах; на короткой задаче — оверхед. Риск cargo-cult, если артефакты не читаются.
5. **Все роли на `model: opus` — нет тиринга.** test/tech-writer могли бы идти на Sonnet/Haiku;
   для OSS/командного сценария стоимость-за-эпик сейчас максимальная по умолчанию.
6. **Флоу не инструментирован** — нет метрик о себе (review-rejection rate, число fix-циклов на
   эпик, cycle-time беда, частота ложных «done»). Механизмы ловли есть, измерения — нет.

### По заявлению «ни один конкурент такого не даёт»
- **Правда:** со-дизайн «граф-как-субстрат-контекста» + «ролевой флоу» + «граф/доки, поддерживаемые
  этим флоу» — необычный замкнутый контур; Backstage/Swimm/Cursor opinionated мультиагентный процесс,
  проводной к собственному арх-графу, не поставляют.
- **Overclaim-риск:** «мультиагентный dev-флоу» как категория в 2026 горячая и переполненная (сам
  Claude Code, экосистема beads, десятки оркестраторов). Как оркестрация флоу хорош, но не уникален.
  Уникальна привязка к Beadloom-субстрату.
- **Защищаемая рамка:** НЕ «у нас лучший флоу» (копируемо, тесно), а **«Beadloom — субстрат
  архитектурной правды, держащий ЛЮБОЙ мультиагентный флоу внутри твоей архитектуры»**. Флоу —
  reference-имплементация и доказательство, не сам товар.

### Стратегические рекомендации
1. **Продавать субстрат, флоу — демо.** Детерминированный `ctx/why` как контекст агента;
   `lint/sync-check/ci` как единый гейт схождения агентов и людей; граф как shared source of truth.
2. **Зашипить флоу как шаблон — `beadloom init --agentic-flow`** (скаффолд `.claude/`; механика
   `config-check`/`setup-rules` уже есть). On-ramp adoption + демо субстрата одновременно; честно —
   отлажен на 46 эпиках. **Сильная killer-feature.**
3. **Инструментировать флоу** — bead cycle-time, review-rejection rate, fix-cycle count на эпик → в
   дашборд (ещё один honest-by-construction showcase).
4. **Тиринг моделей по ролям** (Opus для dev/review, Sonnet/Haiku для test/tech-writer) — снизит
   стоимость-за-эпик.
5. **Честно отделить «solo-fleet» от «team»** — либо строить параллельных владельцев + роль-RBAC +
   разрешение конфликтов между людьми, либо позиционировать как «solo/lead + AI fleet» (само по себе
   горячая ниша 2026).

**Итог §7:** как инженерия процесса — сильно (8/10), связка с субстратом редкая. Но «флоу» сам по
себе — копируемый Markdown на двух чужих субстратах, откалиброван под n=1; он не моат. Моат —
**Beadloom как субстрат архитектурной правды под флоу**. Самый сильный ход: упаковать флоу в
`beadloom init --agentic-flow` (on-ramp + демо) и продавать субстрат, а не оркестрацию.

## 8. Идея: упаковать флоу в MCP Beadloom (Python)

> Добавлено 2026-06-04 по запросу владельца. Идея: перенести мультиагентный флоу (§7) в MCP-сервер
> Beadloom (`services/mcp_server.py`, уже 1026 строк / 14 read-context tools), чтобы он стал
> неотъемлемой частью продукта, а не «копируемым Markdown».

**Вердикт:** направление правильное (tool-agnostic дистрибуция + enforceable tools — реальный
апгрейд над markdown), **но «весь флоу» в MCP не влезает** (оркестрация — это харнесс), и сам по
себе MCP-перенос моатом не становится (моат остаётся субстратом — графом).

### Технический crux: MCP отдаёт инструменты, не оркестрирует
MCP-сервер — провайдер трёх примитивов (**tools** / **resources** / **prompts**); он **вызывается
агентом, но не управляет агентом**. У него нет модели, нет способа породить суб-агента (`Agent()` —
инструмент хоста Claude Code, не MCP), нет влезания в main loop как контроллер. А сердце флоу —
координатор, **порождающий суб-агентов из main loop** — принципиально не может жить в MCP.
(MCP-«sampling» даёт одиночный completion через клиента, не суб-агента с изолированным
контекстом/scoped-tools, и поддержка сырая — оркестрацию ролей не заменяет.)

### Что реально переезжает — три корзины
1. **Детерминированный стейт/скаффолдинг → MCP tools «с зубами» (главный выигрыш).** Превращает
   «модель интерпретирует markdown + bash» в гарантированно-корректный тестируемый Python:
   - `task_init(type, key)` — папка + скаффолд PRD/RFC + **валидный bead-DAG** (4 роли, зависимости) + status-lifecycle.
   - `plan_waves` / `next_ready` — детерминированно вычислить волны/критпуть из DAG (спавн остаётся в харнессе).
   - `bead_context(bead)` — ctx+why+CONTEXT.md+**активные правила** одним вызовом (здесь субстрат даёт зубы).
   - `checkpoint()` / `complete_bead()` — каноничная запись; `complete_bead` **прогоняет lint/sync-check/tests и ОТКАЗЫВАЕТ**, если красно.
   Вот это и есть «integral part, а не копируемый markdown» — для детерминированной части.
2. **Роль-персоны + плейбук координатора + шаблоны → MCP prompts (дистрибуция).** Переносит
   `.claude/commands/*` в версионируемые, шипающиеся с продуктом, tool-agnostic промпты (любой
   MCP-клиент). Но по природе это **всё ещё промпты** — меняется дистрибуция, не природа.
3. **Остаётся в харнессе:** `Agent()`-спавн / main-loop-координатор + **регистрация subagent-типов**
   (`.claude/agents/*` читает Claude Code; MCP не регистрирует `subagent_type`). Персоны можно отдать
   как prompts, но к Agent-tool они так не подключатся.

### Достигает ли цели «не просто markdown»?
- **Enforceability ↑** — инструмент может *отказать* (`complete_bead`-гейт), сильнее markdown.
  Оговорка: модель всё равно решает, звать ли tool → MCP-гейт «advisory-strong»: сильнее markdown,
  слабее CI. Настоящий enforcement остаётся `beadloom ci` (принцип 7 STRATEGY-3).
- **Distribution ↑ / tool-agnostic ↑** — **самый сильный аргумент ЗА.** Сейчас флоу
  Claude-Code-специфичен (`.claude/`); MCP — кросс-IDE стандарт (Linux Foundation, 97M). Tools +
  prompts через MCP = тот же флоу в Cursor/Continue → напрямую обслуживает vision «tool-agnostic,
  one context for everyone».
- **Моат сам по себе НЕ создаёт** — MCP-сервер тоже OSS/копируемый; `task_init`-tool перепишут за
  день. Не копируется привязка к графу Beadloom. Вывод как в §7: MCP усиливает дистрибуцию +
  enforceability, **моат остаётся субстратом**.

### Риски
1. **Воевать с харнессом** — Claude Code сам наращивает агентные фичи. Граница: MCP даёт
   планирование+стейт+гейты, харнесс владеет спавном (симметрично coordinator-context-boundary).
2. **Зрелость MCP-prompts неравномерна** по клиентам в 2026 (tools поддержаны лучше).
3. **Capacity** (#1 риск) — поддержка MCP-поверхности против движущегося spec + движущегося харнесса.

### Рекомендация (конкретно)
Инкремент на существующем `mcp_server.py`, по приоритету ценности:
1. **Сначала 3-4 tool «с зубами»:** `task_init`, `bead_context`, `complete_bead` (гейт с отказом),
   `checkpoint`. Максимальный leverage, всё тестируемо, всё на субстрате.
2. **Потом prompts** для tool-agnostic дистрибуции персон/плейбука (когда поддержка устаканится).
3. **Оркестрацию оставить харнессу** + параллельно `beadloom init --agentic-flow` (§7) для
   Claude-Code-native. Не конфликтуют: MCP — tool-agnostic путь, скаффолд — нативный.

**Рамка позиционирования:** не «флоу теперь внутри MCP» (оркестрации там нет), а **«Beadloom MCP
даёт агенту детерминированные process-tools поверх архитектурного графа — task-init, role-context,
gated-complete — в любом MCP-клиенте»**. Честно, технически точно, усиливает субстрат.
