# AI tech-writer — getting started

> Generated by `beadloom setup-ai-techwriter`. Re-run that command to refresh
> this guide and the vendored harness; do not hand-edit the scaffolded files.

The **AI tech-writer** keeps your docs honest: on a schedule (or on demand) it
finds docs that have drifted from the code (`beadloom sync-check`), has a Goose
agent (Qwen3.7-Plus) rewrite each drifted doc against the current code, proves
freshness with Beadloom's own gate, and opens a **pull/merge request for human
review**. It never auto-merges — `sync-check → 0` proves *freshness*, only a
human proves *correctness*.

## What `setup-ai-techwriter` dropped into this repo

- `tools/ai_techwriter/` — the deterministic, platform-agnostic harness
  (**vendored** so the runner needs only `beadloom` + `goose` + python; no extra
  packaging). The CI wrapper invokes it via `python -m tools.ai_techwriter`.
- The CI wrapper for your platform:
  - GitHub: `.github/workflows/ai-techwriter.yml`
  - GitLab: the `ai-techwriter` job in `.gitlab-ci.yml`
- This guide (`docs/guides/ai-techwriter.md`).

## 3-step setup

1. **Register a self-hosted runner on your VPS** — a GitHub Actions runner *or*
   a GitLab Runner — tagged `self-hosted, ai-techwriter`, with **Goose
   installed** and the model provider reachable. This box holds the API key, so
   scope it to this repo and use an ephemeral (clean-checkout) workspace.
2. **Add the `QWEN_API_KEY` secret** — a GitHub repository secret *or* a GitLab
   masked CI/CD variable. It is referenced only by name in the wrapper and
   resolved on the runner; it is never written into the repo. Optionally also set
   `QWEN_BASE_URL` (same way) to point at a workspace-specific MaaS endpoint; when
   unset the harness uses the generic DashScope gateway.
3. **Run `beadloom setup-ai-techwriter --platform <github|gitlab>`** (already
   done if you are reading this), **commit** the scaffolded files, and **enable
   the pipeline** (GitHub: the workflow is on once committed; GitLab: enable the
   schedule / run it manually).

After that the loop runs on schedule / dispatch and opens PRs/MRs. Nothing else
is configured per-repo — the harness + recipe are repo-agnostic and read *this*
repo's own graph + docs.

## Cost & safety

- **Quality first, scope-bounded cost.** Extended thinking stays on; cost is
  bounded by scope (only drifted docs + scoped context) plus generous runaway
  hard caps (max turns / tokens / wall-clock), never by capping reasoning.
- **Small blast radius.** The agent may write only `docs/**`, runs no arbitrary
  shell, and never marks docs synced or merges — those steps are deterministic
  and owned by the harness + the human reviewer.
- **No secrets in logs;** the key lives only on the runner.
